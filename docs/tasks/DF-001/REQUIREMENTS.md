## 2. Original User Prompt

Create an internal Dictionary Service that stores and manages business terms and definitions. Users should be able to create, read, and update terms. Terms must be unique case-insensitively.

## 3. Business Goal

Provide a centralized business dictionary to reduce misunderstanding of corporate terminology across teams, systems, and documentation.

## 4. User Stories

### US-01: Create Term

As a user,  
I want to create a business term with a definition,  
So that other teams can understand its meaning.

### US-02: Read Term

As a user,  
I want to retrieve a business term by name,  
So that I can view its approved definition.

### US-03: Update Definition

As a user,  
I want to update the definition of an existing term,  
So that the dictionary remains accurate.

## 5. Functional Requirements

### REQ-01: Create Term

- The system must allow users to create a new business term.
- The term must have a definition.
- The system must reject duplicate terms case-insensitively.
- The system must return a created term representation after successful creation.

### REQ-02: Read Term

- The system must allow users to retrieve a term by name.
- Term lookup must be case-insensitive.
- The system must return `404 Not Found` if the term does not exist.

### REQ-03: Update Definition

- The system must allow users to update the definition of an existing term.
- The system must return the updated term representation after successful update.
- The system must return `404 Not Found` if the term does not exist.

## 6. Data Constraints

| Field | Required | Type | Constraint |
|---|---:|---|---|
| term | Yes | string | max 100 characters, unique case-insensitively |
| definition | Yes | string | max 1000 characters |
| createdAt | Yes | timestamp | generated by system |
| updatedAt | Yes | timestamp | generated by system |

## 7. Acceptance Criteria

### AC-01: Create Term Successfully

Given a term does not already exist,  
When a user creates it with a valid definition,  
Then the system stores it in the database,  
And returns HTTP `201 Created`,  
And returns the created term representation.

### AC-02: Reject Duplicate Term

Given a term already exists,  
When a user creates the same term with different letter casing,  
Then the system rejects the request,  
And returns HTTP `409 Conflict`.

### AC-03: Read Existing Term

Given a term exists,  
When a user searches for it by name,  
Then the system returns the term and definition,  
And returns HTTP `200 OK`.

### AC-04: Return Not Found For Missing Term

Given a term does not exist,  
When a user searches for it,  
Then the system returns HTTP `404 Not Found`.

### AC-05: Update Existing Term

Given a term exists,  
When a user updates its definition,  
Then the system stores the new definition,  
And updates `updatedAt`,  
And returns HTTP `200 OK`.

### AC-06: Reject Invalid Input

Given a request contains an empty term, empty definition, or a value exceeding the allowed length,  
When the user submits the request,  
Then the system returns HTTP `400 Bad Request`.

## 8. Business Rules

- Terms must be unique case-insensitively.
- Term text is mandatory.
- Definition text is mandatory.
- `createdAt` must be set only once when the term is created.
- `updatedAt` must be set when the term is created and updated when the definition changes.
- Term lookup must be case-insensitive.

## 9. Out of Scope

- Deleting terms.
- Listing all terms.
- Version history for definitions.
- Authentication and authorization.
- Audit log implementation.
- Frontend UI.
- Bulk import or export.

## 10. Assumptions

- The service is internal.
- Authentication and authorization will be handled separately in a later task.
- PostgreSQL is the default database.
- Java 21 and Spring Boot 3.x are the default backend stack.
- The API is synchronous REST.
