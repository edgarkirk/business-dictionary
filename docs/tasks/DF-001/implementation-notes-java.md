## Layer: acceptance

**What you implemented:**
- `pom.xml` ‚Äî Spring Boot 3.3.4, Java 21, dependencies: web, data-jpa, validation, flyway-core, flyway-database-postgresql, postgresql (runtime), springdoc-openapi, h2 (test), spring-boot-starter-test, archunit-junit5.
- `BusinessDictionaryApplication.java` ‚Äî minimal `@SpringBootApplication` scaffold (no business logic).
- `src/main/resources/application.yaml` ‚Äî production datasource pointing at PostgreSQL; Flyway enabled.
- `src/test/resources/application.yaml` ‚Äî H2 in-memory datasource; Flyway disabled; H2Dialect; ddl-auto: none.
- `src/main/resources/db/migration/.gitkeep` ‚Äî placeholder keeping the migration directory tracked.
- `DictionaryApiAcceptanceTests.java` ‚Äî 18 `@SpringBootTest(RANDOM_PORT)` tests via `TestRestTemplate` covering all six acceptance criteria and every documented error path.

**Key decisions:**
- Used `WebEnvironment.RANDOM_PORT` + `TestRestTemplate` for true end-to-end HTTP coverage (includes servlet filters, error handling, serialization).
- H2 is used instead of Testcontainers because Docker is unavailable in this environment.
- Flyway is disabled in test config; the migration SQL will be introduced in the persistence layer.
- AC-04 and AC-05/404 tests additionally assert `"code"` and `"message"` keys in the error body, ensuring they fail properly in RED (Spring's default 404 body does not contain those keys).

**Deviations:** None ‚Äî no production classes were created.

**Known gaps:** Tests for AC-03/AC-05 that involve creating a term first will fail at setup in RED because the POST endpoint does not exist. This is intentional and correct for the RED phase.

## Layer: entity/domain

**What you implemented:**
- `domain/BusinessDictionaryEntry.java` ‚Äî JPA entity mapped to `business_dictionary` table; UUID PK with `@GeneratedValue(UUID)`; `term` (VARCHAR 100), `normalized_term` (VARCHAR 100, unique), `definition` (VARCHAR 1000); `createdAt`/`updatedAt` via `@CreatedDate`/`@LastModifiedDate` + `AuditingEntityListener`; `updateDefinition(String)` mutation method; protected no-arg constructor per JPA spec.
- `config/JpaAuditingConfig.java` ‚Äî `@Configuration @EnableJpaAuditing`; kept separate from `@SpringBootApplication` so `@WebMvcTest` slices won't load it and trigger "JPA metamodel must not be empty".
- `db/migration/V1__create_business_dictionary_table.sql` ‚Äî PostgreSQL migration creating `business_dictionary` table with `TIMESTAMP WITH TIME ZONE` audit columns and the `ux_business_dictionary_normalized_term` unique index on `normalized_term`.
- `BusinessDictionaryEntryTest.java` ‚Äî `@DataJpaTest` with `@Import(JpaAuditingConfig.class)` and `ddl-auto=create-drop` property override (test config sets `none`); 5 tests covering constructor normalization, casing preservation, `updateDefinition`, persistence with auditing, and `createdAt` immutability with `updatedAt` advancement.
- `pom.xml` ‚Äî added Surefire exclusion for `**/acceptance/**` so the full-stack acceptance tests (RED until all layers are done) do not block the per-layer build.

**Key decisions:**
- `normalizedTerm` is set in the constructor as `term.toLowerCase()` ‚Äî no setter exposed; changing the term is out of scope (only definition updates are required).
- Flyway migration uses PostgreSQL-specific `TIMESTAMP WITH TIME ZONE`; Flyway is disabled in the test config so H2 is never asked to parse it.
- `@DataJpaTest` overrides `ddl-auto` inline rather than in a separate test properties file to keep the override scoped to this test only.

**Deviations:** None from ARCHITECTURE.md.

**Known gaps:** Flyway migration runs only in production (PostgreSQL); H2 schema for acceptance/`@SpringBootTest` tests is generated by Hibernate `create-drop` and therefore does not exercise the actual migration SQL.

> **Token usage for entity/domain layer**: 0 input, 0 output, 0 cache read, 0 cache write, 0 LLM calls (retries: 0)
